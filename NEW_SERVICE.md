# (신규) 캐슬 전투 베팅 서비스

## 개요

킹샷 게임 내 "캐슬 전투" 이벤트는 2주마다 진행된다. 본 신규 서비스는 해당 회차의 각 거점을 어느 집결자가 점령할지 사용자들이 **예측 투표**하고 그 집계 현황을 공유하는 시스템이다.

> "베팅"이라는 명칭은 컨셉상 표현이며, 실제 메커니즘은 **단순 예측 투표 + 집계**다. 게임 내 재화 / 보상과 직접 연동되지 않는다.

## 용어 정의

| 용어 | 의미 |
|------|------|
| 회차 | 1회의 캐슬 전투 이벤트 단위. 운영자가 사이트에서 수동 개최 / 마감 |
| 거점 | 베팅 대상이 되는 전투 지점. 캐슬 1개 + 포탑 4개(11시·2시·5시·7시) = 총 5개 |
| 집결자 | 거점 공격 시 파티를 구성하는 핵심 유저. 가장 전투력이 강한 유저가 집결자가 되고 연맹원들은 집결원으로 합류 |
| 집결원 | 집결자에게 합류해 힘을 보태는 일반 연맹원 |
| 자유전투 | 해당 회차에 베팅 대상으로 지정하지 않은 거점. UI 에서 선택 불가 + "자유전투" 표기 |
| 운영자 | 회차 개최 / 마감, 거점 설정, 집결자 등록을 수행하는 사용자. 킹샷 ID 별도 등록으로 권한 부여 |

## 실제 게임 요소

캐슬 전투는 1767 서버의 약 10개 연맹이 참여하여, 1개의 캐슬과 4개의 포탑을 점령하는 이벤트이다.

- 이벤트 시간: 5시간 진행
- 승리 조건
  a. 주어진 시간 동안 캐슬을 점령한 누적 시간이 가장 많은 연맹이 승리한다.
  b. 연속 점령 2시간 30분을 달성한 연맹은 즉시 승리한다.
- 포탑 점령
  a. 다른 연맹이 캐슬을 점령하고 있을 때 포탑을 점령하면, 해당 포탑이 주기적으로 캐슬에 큰 피해를 입힌다.
  b. 포탑을 점령한 연맹이 캐슬을 공격할 경우 파괴력 증가 버프를 받는다.

## 신규 서비스 콘텐츠 설명

1. **운영자가 회차 개최** — 회차 정보를 입력해 신규 회차를 초기화된 상태로 생성한다.
2. **거점별 베팅 진행 여부 설정** — 5개 거점(캐슬, 포탑 4개) 각각에 대해 베팅을 열지 결정한다. 닫힌 거점은 "자유전투"로 표기된다.
3. **참여 연맹 + 집결자 등록** — 열린 거점마다 전투에 참여할 연맹과 각 연맹의 대표 집결자를 등록한다.
4. **사용자 투표** — 사용자는 열린 거점 각각에 대해 등록된 집결자 중 1명에게 1표씩 행사한다.
   - 한 번 투표한 거점은 **변경 불가**
   - 거점마다 서로 다른 집결자에게 투표 가능 (예: 포탑 11시는 집결자 A, 캐슬은 집결자 C)
5. **운영자가 베팅 마감** — 더 이상 새 투표를 받지 않고 집계를 확정한다.
6. **집계 / 적중률 / 아카이브 조회** — 사용자는 회차별 집계 현황, 자신의 투표 이력, 적중률, 과거 회차 아카이브를 확인할 수 있다.

## 기능 요구사항

### 1. 인증 (사용자)

- 킹샷 ID 입력 → 공식 API 응답에서 **서버 번호 확인**
  - 서버 번호가 1767 이 아니면 인증 거부
- PIN 4자리 인증 (KvK 설문과 동일 방식)
- **재사용 자산**: `tile-match-auth` Edge Function, `AuthDialog` 컴포넌트, `pin-auth` 로직

### 2. 권한 (운영자)

- 별도 테이블에 등록된 킹샷 ID 목록 기반으로 운영자 권한 부여 (KvK 설문 운영자 패턴)
- 운영자 권한 범위
  a. 회차 개최 / 마감
  b. 거점별 베팅 진행 여부 설정
  c. 거점별 참여 연맹 + 집결자 등록
  d. (옵션) 회차 종료 후 실제 게임 결과 입력 — 적중률 정의에 따라 필요 여부 결정

### 3. 회차 관리

- 운영자가 **수동 개최** (자동 스케줄 없음)
- 회차 상태 전이: `준비 중` → `투표 진행 중` → `마감`
- 동시에 활성화 가능한 회차 수: **1개** (확정)
- 회차 정보 필드 예시: 회차명(=이벤트 일자), 이벤트 시작 시각, 메모 등

### 4. 거점 / 집결자 설정

- 5개 거점 각각에 대해 베팅 진행 여부 토글
- 베팅 진행 거점에 한해 참여 연맹 + 집결자 등록
  - **거점당 한 연맹이 집결자 N명 등록 가능 (1:N, 확정)**
  - 예: 포탑 11시에 PNX 연맹의 렐라, 이수가 둘 다 집결자 후보
  - 집결자는 회차 / 거점 단위로 등록 (재사용 마스터 X)
- 자유전투 거점: 사용자 UI 에서 카드 비활성화 + "자유전투" 텍스트만 표기

### 5. 투표 (사용자)

- 열린 거점 각각에 대해 1표씩 행사
- 거점별로 등록된 집결자 중 1명 선택
- **한 번 투표하면 변경 불가** — 신중하게 클릭하도록 UI 에 확인 다이얼로그 1단계 권장 (`appConfirm`)
- **부분 투표 허용 (확정)** — 일부 거점만 투표하고 나머지를 비워 두는 것 허용. 투표하지 않은 거점은 "관심 없음" 으로 간주

### 6. 집계 / 적중률

- **집계 현황**: 거점별 집결자 득표수 / 비율을 표시
  - **실시간 공개 (확정)** — 마감 전부터 거점별 득표 현황 노출. 재미 우선
- **사용자별 투표 이력**: 회차 × 거점 × 선택한 집결자
- **사용자별 적중률**: **둘 다 산출 (확정)**
  - (a) **다수파 적중률** — 내가 투표한 집결자가 해당 거점 1위 득표였는지. 게임 결과 무관, 사이트 내 자체 산출
  - (b) **게임 결과 적중률** — 운영자가 회차 종료 후 실제 점령 집결자를 입력하면 그 결과와 일치한 비율 산출

### 7. 회차 아카이브

- 과거 회차 목록 + 회차별 상세 (참여 연맹 / 집결자 / 득표 현황)
- 사용자 본인의 회차별 투표 이력 + 누적 적중률 (시즌 개념 후속 검토)

### 8. 화면 / UI

기능이 확정된 이후 별도 세션에서 화면 단위로 디테일 설계 예정.

## 비기능 / 운영

- **부정 방지**: **별도 차단 안 함 (확정)**. 게임 부캐(별도 킹샷 ID)는 별도 인격으로 인정. "1 킹샷 ID = 1 표" 원칙만 유지. `tile-match-auth` EF 가 동일 킹샷 ID 의 PIN 재등록을 차단하므로 1 ID 다중 가입은 불가
- **운영자 등록 흐름**: **DB 또는 Claude Code 수동 입력 (확정)**. 관리 UI 안 만듦
- **회차 진행 중 거점 / 집결자 변경**: 원칙적으로 금지 — 이미 투표가 진행된 거점은 변경 시 데이터 정합성 문제 발생. 데이터 모델 단계에서 RLS/CHECK 로 강제

## 재사용 자산 (kingshot-1767 shared / PNX 인덱스)

- 인증: `tile-match-auth` EF + `AuthDialog`
- 다이얼로그: `appAlert` / `appConfirm` (투표 확정 시 한 번 더 확인)
- 페이지 간 공유 데이터: `createStore` + sessionStorage TTL (회차 / 거점 / 집결자 정보)
- 리스트 갱신: `patchList` (집계 현황 실시간 갱신 시)
- 사용자 정보 위젯: `LoginedUserInfo`
- 다국어: `data-i18n` / `onLangChange` 콜백 (새 키: 자유전투 / 베팅 진행 중 / 마감 / 적중률 등)

## 데이터 모델

별도 후속 작업으로 권장 안 제시 예정. 필드별 활용처를 주석 형태로 함께 표기한다.

## 확정 사항 (2026-05-29)

- ✅ **적중률 정의** — **둘 다 산출**
  - (a) 다수파 적중률: 내가 투표한 집결자가 해당 거점 1위 득표였는지 (사이트 자체 산출)
  - (b) 게임 결과 적중률: 운영자가 회차 종료 후 입력한 실제 점령 집결자와 일치한 비율
  - 회차 흐름: `투표 시작 → 투표 → 투표 마감 → 게임 결과 입력`
- ✅ **집계 공개 시점** — **실시간 공개**. 마감 전부터 거점별 득표 현황 노출 (재미 우선)
- ✅ **동시 진행 회차 수** — **1개만**
- ✅ **연맹:집결자 카디널리티** — **거점당 한 연맹이 집결자 N명 등록 가능 (1:N)**
  - 예: 포탑 11시에 PNX 연맹의 렐라, 이수가 둘 다 집결자 후보로 등록 가능
- ✅ **부분 투표** — **허용**. 투표하지 않은 거점은 "관심 없음" 으로 간주
- ✅ **부정 / 중복 가입** — **별도 차단 안 함**. 게임 부캐(별도 킹샷 ID)는 별도 인격으로 인정. "1 킹샷 ID = 1 표" 원칙만 유지. 재미 위주 서비스로 위변조 방지 비용 부담 안 짐
- ✅ **운영자 등록 / 제거** — **UI 안 만듦**. 최고 관리자가 DB 또는 Claude Code 통해 수동 입력
